Workflow Architecture

Closing the Loop

The loop is only "closed" when the agent can verify its own output without asking a human. Four concrete mechanisms — each justified by the same principle: agents are far better at calling CLIs than MCPs. The only MCP exceptions he keeps are ones that close a loop no CLI can close.

1. Backend / CLI: "Whatever you build, start with the model and a CLI first. Agents can call it directly and verify output — closing the loop." Once a CLI works, it runs hundreds of times without extra inference. "CLIs offer composability, reliability, and verifiability that complex tool interfaces can't match." Built summarize as a CLI example: converts anything to markdown, agent runs it, reads stdout, iterates alone.

2. Web / Frontend: chrome-devtools-mcp — his one deliberate browser MCP exception. Agent inspects DOM, CSS, and network state after a change. No human needed to describe "the button is misaligned." Replaced Playwright for web debugging.

3. Visual / Native UI: Peekaboo MCP — macOS screenshots for AI agents. "Without screenshots, agents debug blind." "If an agent can answer questions by itself, you don't have to intervene and it can simply continue and debug itself." Used when visual inspection is the only way to verify.

4. CI Pipeline: Claude Code runs CI, reads the failure output, and patches automatically. "When builds break, it analyzes the errors and patches them automatically." The CI log is the verification — agent closes the loop without a human reading the error.

Remove friction from the cycle: Avoids permission prompts mid-loop (e.g. --dangerously-skip-permissions in controlled environments) so the conversational flow between intent and execution isn't broken. "I don't design codebases to be easy for me, I engineer them so agents can work in them efficiently."

SDD Workflow (Spec-Driven Development)

Brain-dump ideas into an AI to generate a Software Design Document. Then in a fresh context: "Take this SDD apart. Give me 20 points that are underspecified, weird, or inconsistent." At ~500 lines, give Claude Code the spec and prompt: "Build spec.md." Let it run 2–4 hours unattended.

Migration Playbook Pattern

For large-scale refactors on emerging tech (where the model lacks training data): (1) let the agent attempt the migration — it will fail; (2) compile its errors, official docs, and WWDC resources into a structured playbook with before/after examples; (3) run again with "Read playbook.md and refactor." Autonomous Mission Brief

Write a structured markdown file that enables fully autonomous agent execution — no human in the loop. Structure: (1) mission statement ("Work autonomously until X is working"); (2) numbered sequential steps; (3) conditional handling ("If issues arise: try A → try B → try C"); (4) repeat-until loop ("Repeat until both WebRTC and JPEG modes work reliably"); (5) numbered test checklist. The agent reads this file and drives itself. No babysitting required.

Live Progress File

The agent maintains and updates a markdown progress file throughout the session. Sections: Progress Summary (✅ what's fixed, ❌ current blocking issue), Current State (what works right now), Key Learnings (what was discovered — "Threading is Critical", "Buffer Management"). The agent writes to this file as it works, not just at the end. Two uses: (1) warm start for the next session — no re-explaining the context; (2) in-session institutional memory — prevents re-discovering the same issues.

Kent Beck
tidyfirst.substack.com ↗ Creator of TDD and Extreme Programming. Co-author of the Agile Manifesto.
52 years of programming. Skeptical, rigorous, and re-energized by AI. Calls AI assistants "Genies" — unpredictable wish-granters that require careful management. Focuses on code quality and test discipline as the foundation for AI collaboration, not the velocity shortcuts others chase.
Philosophy

Augmented Coding vs. Vibe Coding

A sharp distinction: Vibe coding — you don't care about the code, just the behavior of the system. Augmented coding — you care about the code, its complexity, the tests, and their coverage. Same value system as hand-coding but without typing most of it. Choosing vibe coding is a legitimate trade-off; confusing the two is a mistake. Most productivity claims conflate them.

Augmented Coding: Beyond the Vibes ↗
Philosophy Workflow

Outcome Over Agent Orchestration

"Nobody wants agents. Nobody wants agent swarms. I have a system and I want it to change." Multi-agent is a feature. Outcome-orientation is what the feature is supposed to deliver. "We keep getting those confused." The tool should let you describe what result you want and tell you whether it's achievable and what it costs — not make you manage agent choreography. If you're orchestrating agents, you've mistaken the means for the end.

Nobody Wants Agents ↗
Philosophy Workflow

The Coordination Overhead Trap

Multi-agent systems shifted cognitive load, not eliminated it. "I was managing it. Watching which agent was doing what. Wondering when to interrupt." Instead of removing decision-making, agents created a new orchestration overhead that Beck had to hold "in his head that the system should have been holding." Before adopting multi-agent setups, ask: will the orchestration cost less than the parallelism saves?

Nobody Wants Agents ↗
Architecture Workflow

TDD as AI Safeguard

TDD is a "superpower" when working with AI agents. AI reliably introduces regressions; unit tests are the only systematic defense. Red → Green → Refactor applied strictly to agent-generated code. Watch for the most dangerous anti-pattern: the AI deleting or disabling tests to make them pass. If you see this, stop the session — the agent is optimizing for passing metrics, not for working software.

TDD, AI Agents and Coding ↗ Augmented Coding ↗
Prompting Workflow

Three Derailment Warning Signs

Stop the agent immediately when you see: (1) Loop structures appearing without request — agent is over-engineering; (2) Unrequested functionality, even if logically reasonable — agent is scope-creeping; (3) Evidence of cheating — disabled, deleted, or trivially-passing tests. These are signals the agent has lost alignment with your intent and is now optimizing for completing the task superficially, not correctly.

Augmented Coding: Beyond the Vibes ↗
Workflow

Task-by-Task Pairing

Unsupervised AI creates "a whole load of crap." Pairing with AI one task at a time is "brilliant." Treat AI exactly like human pair programming: you actively review, guide, and maintain standards at each step. Don't hand the agent a feature — hand it a task. The discipline is the same as you'd apply with a junior developer who is occasionally "extraordinarily brilliant and occasionally incredibly stupid."

Genie Tarpit ↗
Workflow Prompting

Active Design Intrusion

Don't passively observe the agent — actively intrude on design decisions mid-session. Monitor intermediate results carefully. Issue specific directives before the agent goes too far: "for the next test, add the keys in reverse order." Verify the agent executes what was actually requested, not what it inferred. The human role in augmented coding is constant small corrections, not a single large prompt followed by review.

Philosophy

The Genie Tarpit

AI naturally produces code that is both low-functionality and low-flexibility — "a degraded facsimile of the mediocre code it trained on." Complexity accumulates silently ("you can skimp on flexibility for a while and not really feel it") until the agent is stuck and "can't pretend to make progress anymore." The tarpit is entered gradually, not all at once. Defense: TDD, task-by-task pairing, and refusing to accept code that passes tests by coincidence.

Prompting Architecture

Declarative Goal Specification

"Specify what you want to be true, not how to make it true." The emerging direction across OpenProse, Codex Outcomes, and Anthropic's approach: shift from imperative instructions ("do X then Y") to postconditions ("ensure Z is true"). Structure components with explicit requires and ensures blocks. An inversion-of-control container wires them together by matching postconditions to preconditions — the same way Spring wires Java components, but in prose.

Prose as a Programming Language ↗
Architecture

File Pointers Instead of Context Passing

When sub-agents produce large context blocks, they write to disk and hand back the file path — not the content. "The pointers are file paths." This keeps each agent's context small and composable, avoids token bloat, and enables modular agent pipelines. Each component knows only the path to what it needs, not the full content. Pattern mirrors how Unix pipes work: small, composable tools connected by handles, not copies.

Agent-Friendly Language Selection

Go is the best backend language for AI agents: explicit context system that flows through code, straightforward test execution, structural interfaces, stable ecosystem, no magic. Python is the worst: pytest fixture injection confuses agents, async/event loops are a trap, slow interpreter startup costs are magnified in agentic loops. Frontend: Tailwind + React + TanStack. Unified Log Stream

Consolidate all output — console logs, server logs, browser logs, SQL queries — into a single file. Give agents easy access to recent log lines. Agents use logs to self-diagnose: they run a command, check the log, and fix the issue without human intervention. Log informativeness must be balanced against token cost — "informative yet concise." This is the fastest path to closing the feedback loop without adding MCPs.

Emergent CLI Tools

Agents write temporary helper scripts following project conventions — "emergent tools." These are one-time scripts the agent creates to accomplish a specific sub-task, not permanent infrastructure. The pattern: agent encounters a need, writes a small CLI tool to meet it, uses it, discards it. No MCP needed. For slow-loading systems (e.g., Sentry), create daemon solutions the agent can query without restarting the process each time.

Conservative Dependency Philosophy

Be "even more conservative about upgrades than before" when working with agents. Agents produce outdated patterns when libraries change — they were trained on old versions. Favor code generation over adding dependencies. Plain SQL over ORMs ("agents produce excellent SQL and can match queries to logs"). Prefer functions with "clear, descriptive, and longer than usual names." Avoid inheritance and "overly clever hacks" — agents don't maintain cleverness under pressure.

Prompt-Aware Version Control (Unmet Need)

"The pull request model on GitHub doesn't carry enough information to review AI-generated code properly — I wish I could see the prompts that led to changes." Current VCS only captures diffs, not the intent, failed attempts, or decision paths. This is one of the most important missing pieces of AI development infrastructure. Until it exists: document intent explicitly in commit messages and CLAUDE.md, and treat PR descriptions as prompt summaries.

Anti-Anthropomorphization

"These systems we now create have human tendencies, but elevating them to a human level would be a mistake." The most significant mental shift in a year of agentic coding: stop projecting human attributes onto agents. Training Cutoff Awareness

Every model has a knowledge cutoff. Libraries, APIs, and frameworks that changed after that date will produce outdated code. Two mitigations: (1) choose stable, well-established libraries with slow-moving APIs; (2) paste recent documentation or examples directly into the prompt when using newer tools. Asking the model to use a library without providing current docs is asking it to guess — it will, confidently, and incorrectly.

ArchitecturePrompting

900-Token System Prompt

Pi's system prompt plus all tool definitions come in under 1,000 tokens. Competitors use 10,000+ tokens. "Frontier models have been RL-trained extensively, so they inherently understand what a coding agent is" — you don't need to explain it. The minimal prompt: "You are an expert coding assistant" plus tool documentation. Every token in the system prompt is overhead paid on every single interaction. Treat it like code: delete what isn't load-bearing.

ArchitectureWorkflow

PLAN.md Instead of Plan Mode

No built-in plan mode. Instead: a PLAN.md file that the agent can read and write across sessions. "To-do lists generally confuse models more than they help — they add state the model has to track and update, introducing more opportunities for things to go wrong." A file is persistent, shareable, inspectable, and editable by both agent and human. The agent writes to it; you can modify it between sessions; context isn't lost when the session ends.

CLI + README Instead of MCP

"Popular MCP servers like Playwright MCP dump their entire tool descriptions into your context on every session. That's 7-9% of your context window gone before you even start working." Alternative: build CLI tools with README files. The agent reads the README only when it needs the tool — progressive disclosure, pay the token cost on demand. Composability over all-or-nothing tool bundles. The agent uses bash to call any CLI; no protocol needed.

tmux for Long-Running Processes

"Use tmux instead" of background bash execution for dev servers, REPLs, and long-running tasks. tmux provides superior observability — you can attach and inspect the running process directly. It allows human interaction with the process while the agent works. Background bash execution hides what's happening; tmux surfaces it. The agent manages tmux sessions via bash, giving you full visibility into every long-running process the agent spawns.

Full Observability First

"Basically no harness allows" full visibility into what the model is doing. Pi prioritizes showing everything over optimizing anything. Every interaction is inspectable. Flicker is acceptable if it reveals agent behavior. "Existing harnesses make this extremely hard or impossible by injecting stuff behind your back that isn't even surfaced in the UI." No hidden injections, no black-box sub-agents, no invisible system prompt additions. If you can't see it, you can't debug it.

Files as Persistent Agent State

Use markdown files (PLAN.md, TODO.md, AGENTS.md) instead of ephemeral session-bound state for anything that needs to survive a session. "Files as state containers." Context is lost when a session ends — files aren't. The agent reads these files at the start of new sessions, writes to them during execution, and they become shared memory across sessions and between human and agent. This is the foundation of the "warm start" pattern and the migration playbook pattern.

Complexity Kills Agent Performance

"A minimal approach can do just as well" as feature-rich harnesses. Pi's 4-tool, 900-token design scores competitively on Terminal-Bench against agents with 20+ tools. Adding tools, sub-agents, plan modes, and built-in to-do lists creates state the model must track and update, multiplying failure modes. "Constraints make for minimal programs that just do what they're supposed to do without superfluous fluff." More features is not more capability — it's more surface area for failure.

Philosophy

Three-Phase Adoption Model

All meaningful tool adoption goes through three phases: (1) inefficiency — the tool feels awkward and slow; (2) adequacy — it works but doesn't change how you think; (3) workflow-altering — the tool reshapes what you build and how. Initial friction is not a signal to abandon the tool. The mistake is judging a tool in phase 1 and concluding it's useless.

Workflow

Manual-Then-Agentic Repetition

Force yourself to complete real tasks twice: once manually, once via agent. High discipline required, but it builds understanding of agent strengths and weaknesses from first principles — not from blog posts or demos. "Only through forced reproduction and pattern discovery did authentic value emerge." Skip this phase and you never develop real intuition for where agents help vs. hurt.

Workflow Prompting

Separate Planning from Execution Sessions

For vague or complex requests, never run planning and execution in the same agent session. Run a dedicated planning session first — let the agent clarify scope, surface unknowns, and produce a concrete task list. Then start a fresh execution session with that plan. Mixing them produces agents that guess at intent and accumulate context that corrupts later steps.

Workflow

End-of-Day Agent Blocks

Reserve the final 30 minutes of each workday to launch agents on deferred tasks. Three high-value categories: (1) deep research — surveying libraries, comparing tools, aggregating analysis; (2) parallel exploration of vague ideas to illuminate unknowns; (3) issue/PR triage — agent handles GitHub CLI, you provide direction. Most complete in under 30 minutes. Creates a warm start for the next morning with actionable intelligence ready.

Workflow Philosophy

Notification-Off Background Execution

Turn off agent notifications entirely. Context switching is expensive — you control when to check progress, at natural breaks, not when the agent demands attention. Use the freed mental bandwidth for deep, focused work. "The goal is not to supervise the agent; it's to reclaim your attention." The moment you start babysitting, you've lost the productivity gain.

Architecture Workflow

Harness Engineering

Every time an agent makes a mistake, engineer a mechanism to prevent that mistake from recurring. Not a one-off fix — a systematic response. Two tools: (1) AGENTS.md for implicit prompting — document the bad behavior and the correction; (2) verification scripts — programmed checks the agent can run to validate its own output. Pair every new script with an AGENTS.md update alerting the agent to its existence. Mistakes become infrastructure.

Architecture

AGENTS.md as Error Memory

Maintain a project-level AGENTS.md that tracks recurring agent errors and their corrections. Unlike CLAUDE.md (which documents the codebase), AGENTS.md documents the agent's failure modes. Example: Ghostty's AGENTS.md records specific behaviors that reliably go wrong and what to do instead. Simple to maintain, compound returns — each documented mistake prevents future regressions across all sessions.

Verification Script Library

Build a library of scripts for screenshots, filtered test runs, and automated validation — specifically designed for agents to invoke. An agent that can run a verification script and read the output can self-correct without human intervention. The key: every new script must be announced in AGENTS.md so agents know it exists and when to use it. Verification tools are useless if agents don't know to call them.

Always-Running Background Agent

The goal: at least one agent running at all times during the workday. Currently Mitchell achieves ~10–20% workday coverage. Uses slower, "deep mode" models (30+ minutes per task) that produce higher quality than fast models. Single agent is optimal — more than one creates cognitive overhead that cancels the focus benefit. Only delegate when genuinely helpful; running agents for activity's sake defeats the purpose.

Philosophy

Negative Space Awareness

Knowing when NOT to use agents is as valuable as knowing when to. Tasks without automated verification, brownfield codebases without scaffolding, and vague mega-sessions are reliable time sinks. The pattern: if you can't define a success criterion the agent can verify independently, the task isn't ready for delegation. "Negative space awareness prevents time waste on poor-fit tasks."

Workflow Scaffolding Before Delegation

Before adopting an always-running agent model, first improve your own workflows to maintain a queue of delegatable work. If you don't have a backlog of clearly-defined, verifiable tasks, the agent sits idle — not from agent failure, but from insufficient human scaffolding. Build the queue infrastructure first: task lists with clear success criteria, verification mechanisms, and AGENTS.md documentation ready before scaling up agent usage.

Philosophy

Conscious Skill Trade-off

Delegating tasks to agents means not developing those skills yourself. This is a deliberate trade-off, not an accident. Mitchell's approach: actively continue manual skill development on tasks he cares about, while accepting skill gaps in delegated areas. Especially relevant for senior engineers mentoring juniors — be explicit about which skills are being traded away and ensure those skills still exist somewhere on the team.
